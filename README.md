# X → Telegram: автопостинг с переводом (бесплатно, без VPS)

Тянет новые посты с аккаунтов X, переводит EN → RU и публикует в твой Telegram-канал.
Работает на **GitHub Actions** — это бесплатный «сервер по расписанию» в облаке.
Платить не нужно, твоё местоположение и проблемы с оплатой роли не играют.

**Почему не будет спама старыми постами и банов, как было на VPS:**
- скрипт хранит ID последнего отправленного поста (`state.json`) и шлёт только новое;
- на первом запуске посты не отправляются — только запоминается точка отсчёта;
- запросов к X мало и без логина → банить практически не за что.

---

## Что понадобится (всё бесплатно)
1. Аккаунт на GitHub.
2. Telegram-бот (создаётся за минуту).
3. Твой канал, куда бот будет постить.

---

## Шаг 1. Создай Telegram-бота
1. В Telegram напиши боту **@BotFather** → команда `/newbot`.
2. Задай имя и username бота. В ответ получишь **токен** вида
   `123456789:AAH...` — сохрани его, это `TELEGRAM_BOT_TOKEN`.

## Шаг 2. Добавь бота в канал и узнай chat_id
1. Открой свой канал → **Управление каналом → Администраторы → Добавить** → добавь своего бота
   (достаточно права «Публикация сообщений»).
2. `TELEGRAM_CHAT_ID` можно указать двумя способами:
   - **Просто `@имя_канала`** (если канал публичный) — самый лёгкий вариант.
   - Или числовой ID вида `-100xxxxxxxxxx` (для приватных каналов). Чтобы его узнать:
     опубликуй любой пост в канале, перешли его боту **@userinfobot** или открой
     `https://api.telegram.org/bot<ТОКЕН>/getUpdates` — там будет `chat.id`.

## Шаг 3. Создай репозиторий на GitHub
1. На github.com → **New repository** → назови, например, `x-to-telegram`.
   Можно сделать **приватным** — Actions работает и в приватных репо бесплатно.
2. Загрузи в репозиторий все файлы из этой папки, сохранив структуру:
   ```
   forward.py
   requirements.txt
   state.json
   .github/workflows/forward.yml
   README.md
   ```
   (Через сайт: **Add file → Upload files**. Папку `.github/workflows` создавать
   вручную не нужно — просто перетащи файл, GitHub создаст путь сам, либо набери
   путь `.github/workflows/forward.yml` в имени файла при создании.)

## Шаг 4. Пропиши секреты и переменные
В репозитории: **Settings → Secrets and variables → Actions**.

Вкладка **Secrets** → New repository secret (это приватные данные):
| Имя | Значение |
|-----|----------|
| `TELEGRAM_BOT_TOKEN` | токен из BotFather |
| `TELEGRAM_CHAT_ID` | `@твой_канал` или `-100...` |

Вкладка **Variables** → New repository variable (это настройки):
| Имя | Значение | Пример |
|-----|----------|--------|
| `ACCOUNTS` | username'ы аккаунтов X через запятую, без `@` | `FabrizioRomano,David_Ornstein` |
| `TRANSLATE` | включить перевод | `true` |
| `SRC_LANG` | язык источника | `en` |
| `DST_LANG` | язык канала | `ru` |
| `SOURCE_MODE` | источник данных | `syndication` |

`RSS_FEEDS` пока оставь пустым (нужен только для запасного режима — см. ниже).

## Шаг 5. Включи и проверь
1. Вкладка **Actions** → если просит — нажми «I understand my workflows, enable them».
2. Выбери workflow **X to Telegram** → **Run workflow** (запуск вручную).
3. Открой лог запуска. На **первом** прогоне ты увидишь строки вида
   `первый запуск — запомнил ID ..., посты не отправляю`. Это правильно: в канал
   ничего не упало, точка отсчёта зафиксирована.
4. Дальше каждые ~5 минут будут прилетать только **новые** посты. Можешь не трогать —
   расписание уже работает.

Готово. Дальше всё крутится само.

## Про задержку
Cron в GitHub срабатывает раз в 5 минут, но при нагрузке запуск иногда задерживается
на несколько минут — реально получается примерно 2–10 минут. В среднем попадаешь в свои
2–7 мин.

**Если нужна стабильно меньшая задержка** — сделай так, чтобы скрипт внутри одного запуска
крутился в цикле. Замени в `.github/workflows/forward.yml` шаг `Run forwarder` на цикл,
который вызывает `python forward.py` каждые ~2,5 минуты в течение ~9 минут:

```yaml
      - name: Run forwarder (loop)
        env:
          # ... те же env, что и выше ...
        run: |
          for i in 1 2 3 4; do
            python forward.py || true
            git add state.json && git commit -m "state [skip ci]" 2>/dev/null && git push || true
            [ $i -lt 4 ] && sleep 150
          done
```
Тогда фактический интервал опроса ~2,5 мин, и всё ещё бесплатно. (Этот вариант чуть
сложнее — бери, только если штатной задержки реально не хватает.)

## Если syndication-источник перестал отдавать посты
Доступ к X — самое хрупкое место в 2026-м; эндпоинт могут прикрыть. Тогда переключись
на RSS как источник, не меняя остальную схему:
1. Сделай RSS-фид для каждого аккаунта — например на **rss.app** (бесплатно) или на
   своём **RSSHub**.
2. В **Variables** поставь `SOURCE_MODE` = `rss` и добавь переменную
   `RSS_FEEDS` вида:
   `FabrizioRomano=https://rss.app/feeds/xxxx.xml,David_Ornstein=https://rss.app/feeds/yyyy.xml`
3. Учти: на бесплатном rss.app фид обновляется медленно (до ~1 часа) — это собьёт
   быструю доставку. Для скорости лучше syndication или свой RSSHub.

## Если что-то не так
- **В канал ничего не приходит после первого запуска** — это by design (первый прогон
  только запоминает ID). Жди новый пост в источнике.
- **Ошибка `Telegram отклонил сообщение`** — проверь, что бот добавлен в канал
  администратором и `TELEGRAM_CHAT_ID` указан верно.
- **`не удалось получить посты`** — источник временно недоступен; следующий запуск
  попробует снова. Если повторяется долго — переключись на RSS (см. выше).
- **Хочешь сбросить и начать заново** — отредактируй `state.json`, сделай его `{}`.
- **Сменить аккаунты** — поменяй переменную `ACCOUNTS`, перезапускать ничего не надо.

> Дисклеймер: автоматический сбор постов из X — серая зона относительно правил X.
> Режим щадящий (редкие запросы, без логина), но риск ограничений со стороны X
> полностью исключить нельзя.
